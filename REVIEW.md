## Part 1: Codebase Walkthrough

---

### 1.1 The Big Picture

Validity is an agentic claim-verification system. You paste a paragraph of text — a news excerpt, a social media post, a product claim, anything — and the system automatically breaks it into individual factual statements, searches the web for both supporting and contradicting evidence, assesses the credibility of each source, and returns a structured verdict for every claim plus an overall assessment for the paragraph as a whole.

The end-to-end flow from a user's perspective: the user pastes a paragraph into a text box and clicks "Verify Claims." The right panel immediately starts showing a live feed of everything the agent is doing — "Decomposing text... found 5 claims," "Searching: 'global GDP growth 2024'," "Source classified: reuters.com → MID tier." A modal then pops up showing the ranked list of claims the agent extracted, asking the user to approve which ones are worth verifying. After the user confirms, the agent continues — searching for evidence, weighing it, assigning verdicts — all visible in real time. When it finishes, the left panel shows structured results: each claim with its verdict badge (HIGH / MEDIUM / LOW / CONTRADICTED), confidence score, and the sources used to reach that verdict, color-coded by credibility tier.

What makes this interesting from an engineering standpoint is the architecture behind it: a multi-node LangGraph agent graph, a bidirectional WebSocket streaming layer, a Human-in-the-Loop pause/resume mechanism built on asyncio primitives, and the same pipeline also exposed as MCP tools for Claude Desktop.

---

### 1.2 The Agent Graph (LangGraph)

**What LangGraph is and why it was chosen**

LangGraph is a library for building stateful, multi-step agent workflows as directed graphs. Each node is a function that reads from a shared state object and writes updates back to it. Edges connect nodes, and conditional edges allow routing to different nodes based on state values.

The alternative to LangGraph would be a plain async Python pipeline — a sequence of `await` calls. That works for a fixed linear chain, but Validity needs things that are awkward to express that way:

- **Conditional routing** after the HITL node (route to `query_gen` if there are approved claims, or to a `no_claims` short-circuit if the user approved zero)
- **A clean state schema** that every node reads from and writes to, with no hidden variable passing through function arguments
- **Async-native execution** that correctly handles both sync and async node functions in the same graph
- **A graph that can be explained visually** — the LangGraph structure directly matches the architecture diagram

LangGraph represents the pipeline as a `StateGraph(VerificationState)`, where `VerificationState` is a TypedDict. Each node receives the full state and returns a dict of updates to merge back in. This TypedDict is the central data contract of the entire pipeline.

**The VerificationState schema** (`backend/agents/state.py`)

```
input_text          — the paragraph the user submitted
claims              — raw claims extracted by decompose
ranked_claims       — top N claims scored and selected by rank
approved_claims     — the subset the user approved in HITL (or all ranked in skip mode)
search_queries      — affirm + refute queries per claim, generated by query_gen
search_results      — raw results from the search API, tagged with claim_id + intent
classified_results  — same results with source_tier added by classify
evidence_assessments — per-source LLM assessments: SUPPORTS / CONTRADICTS / IRRELEVANT
claim_verdicts      — per-claim verdict + confidence + supporting/contradicting evidence
overall_verdict     — the synthesized final verdict object
run_id              — UUID for this run (used to look up the callback handler in nodes)
errors              — list of error strings appended by any failing node
```

Every node reads what it needs and writes only the keys it owns. No node touches another node's outputs.

**Graph structure** (`backend/agents/graph.py`)

```
START
  │
  ▼
decompose
  │
  ▼
rank
  │
  ▼
hitl ─────[approved_claims is empty]──→ no_claims ──→ END
  │
  [approved_claims non-empty]
  │
  ▼
query_gen → search → classify → weigh → verdict → synthesize → END
```

The conditional router `_route_after_hitl` checks `state.get("approved_claims")` and returns either `"query_gen"` or `"no_claims"`. The `_no_claims_node` short-circuits with a minimal "low" overall verdict with `total_claims=0`, so the pipeline completes cleanly and emits `pipeline_complete` — no crash, no hanging WebSocket.

The graph is compiled with `graph.compile()` — no `MemorySaver` checkpointer. This is a deliberate decision explained in section 1.3.

**Node deep dives**

#### Node 1: decompose (`backend/agents/nodes/decompose.py`)

**What it does:** Calls the LLM with a system prompt instructing it to extract every atomic, verifiable factual claim from the input text. "Atomic" means each claim is a single self-contained statement. "Verifiable" means it could in principle be checked against evidence — opinions, predictions, and normative statements are explicitly excluded.

**LLM:** GPT-4o (`complexity="high"`). This is the most important call in the pipeline. If decomposition is wrong — groups two claims together, lets opinions through, misses a key claim — every downstream node works with bad input. A weaker model produces noisy claims that cascade errors. GPT-4o's stronger comprehension is the right trade-off even at higher cost.

**Reads from state:** `input_text`, `run_id`

**Writes to state:** `claims` — a list of dicts, each with a UUID `id`, `text`, and `importance_score` initialized to `0.0`

**Interesting detail:** The node handles JSON output defensively. If the LLM wraps its JSON in markdown code fences (which models sometimes do), the node strips them before parsing. This pattern is repeated in every LLM node in the pipeline — prevents `json.loads` failures from breaking anything.

**Error handling:** On any exception, appends to `errors` and returns an empty `claims` list. The pipeline continues with no claims, eventually producing a "no claims verified" verdict rather than crashing.

---

#### Node 2: rank (`backend/agents/nodes/rank.py`)

**What it does:** Takes the raw claims from decompose and asks the LLM to score each one on two dimensions: *verifiability* (0–1: how easily can this be checked with web searches?) and *importance* (0–1: how central is this claim to the text's meaning?). The combined score is their average. Claims are sorted descending, and the top `MAX_CLAIMS` (default 5, from `.env`) are kept as `ranked_claims`.

**LLM:** GPT-4o-mini (`complexity="standard"`). Ranking against clear criteria with structured JSON output is exactly the kind of task mini handles well. No deep reasoning required.

**Reads from state:** `claims`, `run_id`

**Writes to state:** `ranked_claims` — same claim objects with `importance_score` updated from the LLM's `combined_score`

**Important design note:** Rank writes only `ranked_claims`, not `approved_claims`. That field is left exclusively for the HITL node. This keeps two concepts clean: "what the LLM thinks is worth verifying" versus "what the user actually approved."

**Fallback:** If the LLM call fails, rank falls back to the original decompose order, capped at `MAX_CLAIMS`. The pipeline never gets stuck because ranking failed.

---

#### HITL Node (`backend/agents/nodes/hitl.py`)

This is an `async def` node — the most architecturally interesting one. Full details in section 1.3. Short summary here:

**What it does:** Pauses the pipeline for human review. Emits the ranked claims to the frontend via the callback queue, then awaits an `asyncio.Event` until the user sends back their approved list via WebSocket.

**LLM:** None. Pure asyncio coordination.

**Reads from state:** `ranked_claims`, `run_id`

**Writes to state:** `approved_claims` — the user-approved subset, or all `ranked_claims` in skip mode

**Skip mode:** If `cb is None` (no callback registered — sync endpoint or MCP path) or `cb.hitl_event is None` (callback present but HITL disabled), the node immediately returns `{"approved_claims": ranked_claims}`. Same code, same graph, no HITL pause needed.

**Timeout:** After 300 seconds with no response, auto-approves all ranked claims and continues. The pipeline never hangs forever.

---

#### Node 3: query_gen (`backend/agents/nodes/query_gen.py`)

**What it does:** For every approved claim, generates two sets of search queries — *affirm* queries designed to find supporting evidence, and *refute* queries designed to find contradicting evidence. The system prompt calls this "adversarial search query generation" and explicitly states that trying to disprove a claim is what makes verification meaningful rather than just confirmation-bias searching.

**LLM:** GPT-4o-mini (`complexity="standard"`). Understanding a claim's semantics well enough to write good counter-queries is structured enough for mini.

**Reads from state:** `approved_claims`, `run_id`

**Writes to state:** `search_queries` — a flat list of `{claim_id, intent: "affirm"|"refute", query}` objects for all claims combined

**Key design decision:** The adversarial search design. For "The Great Wall of China is visible from space," affirm queries look for evidence it's true; refute queries search specifically for debunking — "Great Wall China NOT visible from space myth." Without refute queries, the pipeline could find low-quality sources repeating the myth and miss the NASA articles that explicitly disprove it.

---

#### Node 4: search (`backend/agents/nodes/search.py`)

**What it does:** Executes all search queries concurrently and returns tagged results. Uses `asyncio.gather(*all_tasks)` to run every query simultaneously — for a 5-claim run with ~4 queries per claim, that's ~20 concurrent HTTP requests. Results are tagged with `claim_id` and `query_intent`, then deduplicated by URL within each claim group.

**LLM:** None. Pure HTTP calls to the search API.

**Reads from state:** `search_queries`, `run_id`

**Writes to state:** `search_results` — list of `{url, title, snippet, claim_id, query_intent, source_tier: None}`. The `source_tier` field is initialized to `None` here and filled by the classify node.

**Async node:** Search makes real HTTP calls so it's `async def`. The inner `run_query` coroutine is created once per query, and all are gathered simultaneously.

**Error isolation:** `asyncio.gather(..., return_exceptions=True)` catches individual query failures without killing the others. Failed queries are logged as warnings and skipped. The pipeline continues with whatever results were successfully retrieved.

**Streaming events:** Emits a "Searching: 'query' (affirm)" event per query and a "Found: 'title' (domain)" event per result. This is why the ThoughtPanel is densest during the search phase.

---

#### Node 5: classify (`backend/agents/nodes/classify.py`)

**What it does:** Assigns a credibility tier to every search result URL using pure domain-matching heuristics. No LLM call, no external API — instantaneous and free.

**LLM:** None. The only "intelligence" in the pipeline that doesn't touch an LLM.

**Reads from state:** `search_results`

**Writes to state:** `classified_results` — same as search_results but with `source_tier` set to `"high"`, `"mid"`, or `"low"`

**The classification logic:**
- **HIGH tier:** Exact hostname in an 18-domain whitelist (`arxiv.org`, `pubmed.ncbi.nlm.nih.gov`, `nasa.gov`, `nature.com`, `nejm.org`, `nih.gov`, `who.int`, `cdc.gov`, etc.) OR hostname ending in `.gov` or `.edu`
- **MID tier:** Exact hostname in a 26-domain whitelist (`reuters.com`, `apnews.com`, `bbc.com`, `nytimes.com`, `theguardian.com`, `economist.com`, `bloomberg.com`, `snopes.com`, `wikipedia.org`, etc.) OR any `.org` not already matched above
- **LOW tier:** Everything else — general `.com`, unknown domains, vendor sites

**Honest limitations (explicitly documented in spec and README):** A `.edu` professor's personal blog gets HIGH tier the same as a peer-reviewed journal. A `.com` investigative piece gets LOW even if it's excellent. Domain age, author credentials, citation count — none of this is captured. The spec calls this a v1 limitation and puts an ML-based classifier on the v2 roadmap. The key is that this is acknowledged, not hidden.

**Per-source streaming events:** Classify emits `"Source classified: nasa.gov → HIGH tier"` for every single URL. This is what makes the ThoughtPanel's classify phase feel impressively active during a demo.

---

#### Node 6: weigh (`backend/agents/nodes/weigh.py`)

**What it does:** For each approved claim, takes all that claim's classified search results and asks the LLM to assess each source's snippet: does it SUPPORT, CONTRADICT, or is it IRRELEVANT to this specific claim? One LLM call per claim, passing all of that claim's sources in one message.

**LLM:** GPT-4o (`complexity="high"`). This is the second most important call. Reading a search snippet and determining whether it supports or contradicts a specific claim requires genuine comprehension — partial support, implicit contradiction, tangential relevance. This is where model quality directly affects verdict accuracy. GPT-4o-mini makes more mistakes here.

**Reads from state:** `classified_results`, `approved_claims`, `run_id`

**Writes to state:** `evidence_assessments` — a flat list of `{claim_id, source, assessment, reasoning, weight}`. The `weight` is applied at this stage from `TIER_WEIGHTS = {"high": 1.0, "mid": 0.6, "low": 0.3}`. A high-tier source contradiction carries 3.3x more weight than a low-tier source claiming support.

**Error isolation per claim:** If the LLM call fails for a particular claim, all of that claim's sources are marked `irrelevant`. The claim still gets a verdict; the pipeline doesn't crash.

---

#### Node 7: verdict (`backend/agents/nodes/verdict.py`)

**What it does:** For each approved claim, passes the weighted evidence summary to the LLM and asks it to assign a verdict: HIGH, MEDIUM, LOW, or CONTRADICTED, plus a confidence score (0–1). One LLM call per claim.

**LLM:** GPT-4o-mini (`complexity="standard"`). The hard reasoning is already done by the weigh node. Assigning a verdict from a clear, pre-processed evidence summary with defined rules is a structured task — mini is appropriate and keeps costs down.

**Reads from state:** `evidence_assessments`, `approved_claims`, `classified_results`, `run_id`

**Writes to state:** `claim_verdicts` — one entry per claim with `claim_id`, `claim_text`, `verdict`, `confidence`, `supporting_evidence`, `contradicting_evidence`, and `sources`

**Verdict definitions in the system prompt:**
- `"high"` — strong weighted support from credible sources, no significant contradictions
- `"medium"` — mixed evidence, only low-tier sources, or insufficient evidence
- `"low"` — weak or no support, or contradicted by credible sources
- `"contradicted"` — high-tier or mid-tier sources directly contradict the claim

**Error isolation:** If the LLM call fails for a specific claim, that claim defaults to `verdict="medium"` and `confidence=0.3`. Other claims are unaffected.

---

#### Node 8: synthesize (`backend/agents/nodes/synthesize.py`)

**What it does:** Takes all per-claim verdicts and synthesizes them into a single overall verdict for the paragraph, with a 2-3 sentence natural language summary. One LLM call total.

**LLM:** GPT-4o-mini (`complexity="standard"`). Summarizing a list of claim verdicts into an overall assessment is structured and low-complexity.

**Reads from state:** `claim_verdicts`, `approved_claims` (for importance scores), `run_id`

**Writes to state:** `overall_verdict` — the final result object: `summary`, `verdict` (high/medium/low/mixed), `claim_verdicts` (passed through), `total_claims`, plus four verdict-type counts

**Heuristic fallback:** If the LLM call fails, synthesize falls back to a rule: if any claim is contradicted → "mixed"; if ≥70% of claims are high → "high"; if ≥50% are low → "low"; otherwise → "medium". The pipeline always produces a verdict.

**Important:** The entire final result — overall verdict plus all per-claim details, sources, and evidence — is nested inside the single `overall_verdict` key. The FastAPI layer reads this one key and stores it in the run store.

### 1.3 The HITL Flow

**What HITL means and why it exists**

HITL stands for Human-in-the-Loop. It's a named pattern in agentic system design, not a UX nicety. The problem is the "claim explosion": decomposing a paragraph typically yields 8-15 claims, many trivial — "The meeting was held on Tuesday," "He is a professor at Harvard." Verifying all of them wastes search API calls (every claim generates 4-6 queries), wastes LLM tokens (every claim gets evidence weighing and a verdict call), and produces an overwhelmingly verbose result.

The HITL pause after Node 2 gives the user a focused moment to prune the list. They see the LLM's ranked claims, remove trivial ones, and can add claims the LLM missed. The pipeline then runs only on a focused, user-validated set. This is architecturally analogous to the human approval step in production agentic systems before committing to expensive or irreversible actions.

**The implementation approach — asyncio.Event (Approach B)**

The spec and `hitl.py` docstring document the decision: the graph is compiled with `graph.compile()` — no `MemorySaver` checkpointer. Using LangGraph's built-in `interrupt()` pattern would require compiling with a checkpointer and refactoring the pipeline invocation in `main.py` from `await verification_graph.ainvoke(initial_state)` to a pattern involving `Command(resume=...)` and catching `GraphInterrupt` exceptions — a significant change to working Phase 2 code.

Instead, a simpler mechanism: since `ainvoke()` runs as an asyncio coroutine on the same event loop as the FastAPI WebSocket handler, an `asyncio.Event` is the textbook coordination primitive. The HITL node awaits the event; the WebSocket handler sets it when the user responds. Both live in the same event loop — no threads, no inter-process communication needed.

**The exact sequence of events**

1. `POST /api/verify` creates an `asyncio.Queue` for this run, stores it in `runs[run_id]`, launches `_run_pipeline(run_id, text, enable_hitl=True)` as an asyncio background task, and immediately returns `{"run_id": ..., "status": "running"}`.

2. The frontend receives `run_id`, opens a WebSocket to `WS /api/verify/{run_id}/stream`. The WebSocket handler starts two concurrent async tasks: `send_events()` (drains the queue and sends events to client) and `receive_messages()` (listens for incoming messages from the client).

3. Inside `_run_pipeline`, a `StreamingCallbackHandler` is created with `cb.hitl_event = asyncio.Event()` and `cb.hitl_response = {}`. This handler is registered globally under `run_id` in the `_run_callbacks` dict.

4. The pipeline runs: decompose → rank → hitl.

5. Inside `hitl_node`, it calls `get_callback(run_id)`. Since `cb.hitl_event is not None`, it enters the interactive path — not skip mode.

6. `hitl_node` calls `await cb.aemit({"type": "hitl_request", "data": {"claims": ranked_claims}, ...})`, putting the event on the queue.

7. `send_events()` drains the queue item and calls `await websocket.send_json(item)`.

8. `hitl_node` then reaches `await asyncio.wait_for(cb.hitl_event.wait(), timeout=300)` — suspended, waiting.

9. The frontend's `useVerify.js` receives the `hitl_request` message, sets `hitlClaims` in state, transitions `status` to `"hitl"`. `App.jsx` renders `<ClaimModal>` because `hitlClaims` is non-null.

10. The user reviews claims, toggles checkboxes, optionally adds custom claims, clicks "Verify Selected Claims."

11. `ClaimModal.handleConfirm()` calls `onConfirm(approved)` → `submitHitl(approvedClaims)` → `wsRef.current.send(JSON.stringify({type: "hitl_response", approved_claims: approvedClaims}))`.

12. `receive_messages()` gets the message, sees `type === "hitl_response"`, calls `_handle_hitl_response(run_id, message)`.

13. `_handle_hitl_response` validates claims, writes `cb.hitl_response["approved_claims"] = validated`, then calls `cb.hitl_event.set()`.

14. Back in `hitl_node`, `await asyncio.wait_for(cb.hitl_event.wait(), ...)` resolves. The node reads `cb.hitl_response["approved_claims"]`, validates again (defense in depth), emits "Resuming with N approved claims...", and returns `{"approved_claims": validated}`.

15. The conditional router `_route_after_hitl` sees non-empty `approved_claims`, routes to `query_gen`. Pipeline resumes.

**What happens when HITL is skipped**

Both the sync endpoint (`POST /api/verify?sync=true`) and the MCP server call `ainvoke()` without registering a callback. Inside `hitl_node`, `get_callback(run_id)` returns `None`. The node sees `cb is None` and immediately returns `{"approved_claims": ranked_claims}`. Same graph, same code, no pause.

### 1.4 The Backend (FastAPI)

**API endpoints** (`backend/main.py`)

`GET /api/health` — Returns `{"status": "ok", "search_provider": "...", "llm_provider": "..."}`. Used by Docker's healthcheck and for operational visibility into which providers are active.

`POST /api/verify` — Two modes via `?sync` query param:
- **Default (async):** Creates an `asyncio.Queue`, stores the run in the `runs` dict, launches `_run_pipeline` as a background task via `asyncio.create_task()`, and immediately returns `{"run_id": "...", "status": "running"}`. The client is expected to connect via WebSocket for updates.
- **`?sync=true`:** Runs the pipeline inline (awaiting it directly), no callback registered, HITL skipped. Returns the full `VerifyResponse` when complete. Phase 1 compatibility mode, also used for curl-based testing.

`WS /api/verify/{run_id}/stream` — Bidirectional WebSocket. Validates the `run_id` exists and is running. If the run already completed or errored, sends the final event and closes immediately (replay for late-connecting clients). Otherwise starts `send_events()` and `receive_messages()` concurrently, uses `asyncio.wait(return_when=FIRST_COMPLETED)` to clean up whichever task outlives the other.

`GET /api/verify/{run_id}/result` — Poll fallback. Returns run status and result from `runs` dict. Returns 404 for unknown run_id.

**The in-memory run store**

The `runs` dict at module level is the entire persistence layer for the web app. Each entry:
```python
{
    "run_id": "...",
    "status": "running" | "completed" | "error",
    "queue": asyncio.Queue(),   # events flow from pipeline to WebSocket through here
    "result": {...} | None,     # OverallVerdict dict when completed
    "error": "..." | None,
}
```

This is intentionally simple. Restarting the server loses all run history. Multiple processes wouldn't share state. For a portfolio project, this is the right trade-off — it eliminates Redis/database dependencies and keeps setup to a single `docker compose up`.

**The callback bridge** (`backend/agents/callbacks.py`)

`StreamingCallbackHandler` bridges the LangGraph pipeline to the WebSocket layer. It holds:
- A reference to the per-run `asyncio.Queue`
- The `run_id` (for event stamping)
- A reference to the event loop (for thread-safe cross-thread puts)
- `hitl_event` and `hitl_response` for HITL coordination

Registered in a module-level global dict `_run_callbacks: dict[str, StreamingCallbackHandler]`. Every node calls `get_callback(run_id)` at its start and emits events if a handler exists.

Two emission methods exist for thread-safety reasons:

- `emit()` (synchronous): used by sync nodes. LangGraph runs sync node functions in a `ThreadPoolExecutor` — not the main event loop. `emit()` uses `asyncio.run_coroutine_threadsafe(self.queue.put(ev), self._loop)` to safely cross from the thread into the event loop's queue.
- `aemit()` (async): used by async nodes (`hitl_node`, `search_node`). These run directly in the event loop, so `await self.queue.put(ev)` is safe.

Calling `await queue.put()` from a thread that isn't the event loop would break asyncio silently or crash. The `emit()` / `aemit()` distinction is the correct solution.

### 1.5 The WebSocket Streaming Architecture

**Why WebSocket instead of polling or SSE**

HITL requires bidirectionality: the server sends a `hitl_request` event to the client, and the client sends a `hitl_response` back — on the same logical connection. Server-Sent Events is one-directional (server to client only), which would require a separate POST endpoint for the HITL response and explicit session correlation. HTTP polling would add latency and complexity for every event. WebSocket is the natural choice for a bidirectional, low-latency, stateful streaming connection.

**The event schema**

Every event emitted by the pipeline has this shape:
```json
{
  "type": "node_event" | "hitl_request" | "pipeline_start" | "pipeline_complete" | "pipeline_error",
  "node": "decompose" | "rank" | "hitl" | "query_gen" | "search" | "classify" | "weigh" | "verdict" | "synthesize" | "pipeline",
  "status": "running" | "completed" | "error" | "waiting",
  "detail": "Human-readable description of what just happened",
  "run_id": "...",
  "timestamp": "2024-01-01T00:00:00+00:00",
  "data": { ... }   // optional — e.g. {"claims": [...]} for hitl_request, verdict dict for pipeline_complete
}
```

`run_id` and `timestamp` are added automatically by `StreamingCallbackHandler._make_event()` if missing. Node functions only need to specify `type`, `node`, `status`, `detail`, and optionally `data`.

**Event flow**

```
Node function runs
  │
  ├── sync node (decompose, rank, classify, weigh, verdict, synthesize)
  │     cb.emit(event)
  │       └── asyncio.run_coroutine_threadsafe(queue.put(event), loop)
  │
  └── async node (hitl, search)
        await cb.aemit(event)
          └── await queue.put(event)

asyncio.Queue (per run)
  │
  └── send_events() coroutine  →  await websocket.send_json(item)
        │
        └── Frontend ws.onmessage
              │
              ├── node_event       → setEvents(prev => [...prev, event])
              ├── hitl_request     → setHitlClaims(claims), setStatus('hitl')
              ├── pipeline_complete → setVerdict(data), setStatus('completed')
              └── pipeline_error   → setError(detail), setStatus('error')
```

**The HITL reverse path:**
```
User confirms in ClaimModal
  └── submitHitl(approvedClaims)
        └── ws.send(JSON.stringify({type: 'hitl_response', approved_claims: [...]}))

receive_messages() coroutine
  └── _handle_hitl_response(run_id, message)
        ├── validates claims
        ├── cb.hitl_response["approved_claims"] = validated
        └── cb.hitl_event.set()
              └── hitl_node() resumes from await asyncio.wait_for(...)
```

**The sentinel pattern**

When the pipeline finishes (or errors), `_run_pipeline` puts `_SENTINEL = object()` onto the queue as the last item. The `send_events()` coroutine checks `if item is _SENTINEL: return`, exits cleanly. `asyncio.wait(return_when=FIRST_COMPLETED)` then cancels `receive_messages()` and the WebSocket closes.

### 1.6 The Frontend

**The two-panel layout**

`App.jsx` renders a responsive two-column layout:
- **Left panel (58% on desktop):** InputPanel on top, VerdictPanel below — the "input and output" side
- **Right panel (42% on desktop):** ThoughtPanel, full height — the "transparency" side, live agent reasoning stream

On mobile, the layout stacks vertically. The ThoughtPanel has `minHeight: '500px'` to stay visible.

**useVerify.js — the central state machine**

This hook owns all connection state and exposes a clean interface. Nothing in the UI touches WebSocket directly — everything goes through this hook.

State managed:
- `status` — `"idle" | "running" | "hitl" | "completed" | "error"`. Source of truth for what the UI shows.
- `events` — append-only array of streaming events for ThoughtPanel. Reset on new submission.
- `verdict` — the `OverallVerdict` object, null until `pipeline_complete`.
- `error` — error string, null if none.
- `runId` — current run UUID.
- `hitlClaims` — non-null array of claim objects while HITL modal should be shown; null otherwise.

Refs (don't trigger re-renders):
- `wsRef` — holds the WebSocket instance. A ref prevents re-renders on WS state changes.
- `currentRunRef` — tracks the active run ID. On `ws.onmessage`, the handler checks `if (currentRunRef.current !== newRunId) return` — prevents stale events from a previous run corrupting the UI if the user re-submits before the old WebSocket closes.

`submitText(text)`: POSTs to `/api/verify`, gets `run_id`, opens WebSocket, sets up message handlers.

`submitHitl(approvedClaims)`: Sends `{type: 'hitl_response', approved_claims}` over the open WebSocket, sets `hitlClaims` to null, sets `status` back to `'running'`.

**InputPanel** (`frontend/src/components/InputPanel.jsx`)

Controlled textarea and submit button. `isRunning = status === 'running' || status === 'hitl'` — the textarea is disabled during both these states (user cannot re-submit mid-run). Shows a spinner with "Verifying…" during the run, a search icon and "Verify Claims" at idle. Has no knowledge of WebSocket — receives `onSubmit(text)` as a prop.

**ThoughtPanel** (`frontend/src/components/ThoughtPanel.jsx`)

Live agent reasoning log. Renders a scrolling list of `EventRow` components.

`NODE_COLORS` and `NODE_BG` map node names to distinct Tailwind color classes: decompose = violet, rank = blue, query_gen = cyan, search = emerald, classify = amber, weigh = orange, verdict = rose, synthesize = purple. Visually easy to track which phase is active.

`EventRow` shows: an icon (green checkmark for completed, red X for error, blue pulsing dot for running, lightning bolt for others), the node name in its color padded to 20 chars for alignment, and the detail text.

**Smart auto-scroll:** A `scrollRef` tracks the container. `useEffect` fires on every `events` change. Checks `isUserScrolledUp.current` first — if the user scrolled more than 60px up from the bottom, auto-scroll is suppressed. `handleScroll` keeps the ref current. The panel respects manual scrolling.

**ClaimModal** (`frontend/src/components/ClaimModal.jsx`)

The HITL review interface. Rendered by `App.jsx` when `hitlClaims` is non-null.

Internal state:
- `checked` — `{[claim.id]: boolean}`, all initialized to `true`
- `customText` — the add-claim input field
- `customClaims` — user-added claims, each gets `crypto.randomUUID()` as ID and `importance_score: 1.0`

Each claim row is clickable and toggles its checked state. Checked claims highlighted in blue; unchecked grayed out. Custom claims show a "Custom" badge instead of an importance score.

"Verify X Selected Claims" button disabled when `approvedCount === 0` — can't submit nothing. On click, `handleConfirm()` builds the approved array from checked original + custom claims and calls `onConfirm(approved)` → `submitHitl(approved)`.

**Closing the modal:** `App.jsx` passes `onClose={() => submitHitl(hitlClaims)}` — closing the modal (X button, backdrop click, Cancel) auto-approves all original ranked claims. The pipeline always gets a response; closing the modal never abandons the run.

**VerdictPanel** (`frontend/src/components/VerdictPanel.jsx`)

Renders based on `status`:
- `"idle"` → placeholder text
- `"running"` → blue spinner
- `"hitl"` → amber spinner ("Reviewing claims — pipeline paused…")
- `"error"` → red error card
- `"completed"` → full verdict rendering

Completed verdict: an overall banner (gradient border colored by verdict type via `VERDICT_STYLES`), a stats row with count chips ("3 high", "1 contradicted"), and a list of `ClaimCard` components.

Each `ClaimCard` has local state for two collapsible sections — "N sources" and "Evidence (X supporting, Y contradicting)." Sources render as `SourceCard`: tier badge (teal=high, slate=mid, orange=low), intent badge (green=affirm, rose=refute), title as external link, domain in small gray text. Evidence items render with a check/X/minus icon and the LLM's one-sentence reasoning.

**App.jsx**

Calls `useVerify()` once. Renders header, two-panel layout, conditionally renders `ClaimModal`. No logic of its own — just wires components to the hook.

### 1.7 The MCP Server

**What MCP is and why Validity has one**

MCP (Model Context Protocol) is a standard protocol allowing Claude Desktop and other LLM clients to call external tools. When a Claude Desktop user asks "can you verify these claims?", the MCP integration means Claude calls the Validity pipeline directly and returns results — without the user opening a browser. The Validity MCP server uses FastMCP, a Python framework that wraps async functions as MCP tool definitions. `python mcp/server.py` starts a process that Claude Desktop can discover and call.

**The three tools** (`mcp/server.py`)

`verify_text(text: str) -> str`
Runs the full 8-node graph via `await verification_graph.ainvoke(initial_state)` with no callback registered. HITL is auto-approved (hitl_node sees `cb=None`, returns all ranked_claims). Returns a formatted text verdict string via `_format_verdict()`. Stores result in `_run_store` for `get_run()`.

`verify_text_interactive(text, approved_claim_ids=None, preview_id=None) -> str`
A two-step stateless pattern that approximates HITL within the stateless tool protocol:

*Step 1 (text only):* Calls `decompose_node(state)` and `rank_node(state)` directly (not through the graph). Caches `ranked_claims` in `_run_store[run_id] = {"type": "preview", "ranked_claims": [...]}`. Returns a formatted claim list with IDs and importance scores, plus instructions for Step 2.

*Step 2 (text + approved_claim_ids + preview_id):* Looks up cached claims from `_run_store[preview_id]`, filters to the user-specified IDs, calls `_run_from_approved_claims()` which runs the remaining nodes directly in sequence: `query_gen_node → search_node → classify_node → weigh_node → verdict_node → synthesize_node`. Returns formatted verdict.

This is a smart adaptation: MCP tools are stateless (no persistent connection), so real-time pause/resume is impossible. The two-step design preserves the spirit of HITL — user reviews and approves claims before verification runs — within the protocol's constraints.

`get_run(run_id: str) -> str`
Looks up `_run_store` by run_id. Returns the formatted verdict, a "still in progress" message, or an error message. **Key limitation documented in code and README:** Works only for runs started in this same MCP server process. The FastAPI web server has its own separate `runs` dict — the two processes don't share memory.

**Direct node invocation in `_run_from_approved_claims`**

The MCP server calls node functions directly rather than invoking the full graph. Nodes are plain Python functions that accept a state dict and return an update dict. The MCP server manually builds a state dict, calls nodes in sequence with `state.update(node_function(state))`, and accumulates results. This bypasses the LangGraph graph entirely — it works because nodes have no side effects outside the state dict, and since no callback is registered, there are no streaming events to handle.

**`_format_verdict(verdict_dict) -> str`**

Produces human-readable text suitable for Claude Desktop's response pane: overall verdict label, summary, then per-claim results (claim text, verdict, confidence, sources with tier and contradiction flags). MCP tools return strings, not JSON or HTML.

### 1.8 The Provider Abstraction

**The dual-model pattern**

Not all LLM calls require the same reasoning capability. Validity makes 18 LLM calls per 5-claim run. Six — decomposition and per-claim evidence weighing — require genuine comprehension and nuanced judgment. The remaining twelve — ranking, query generation, per-claim verdicts, and synthesis — are structured tasks with constrained outputs.

`get_llm(complexity: str)` in `backend/config.py` is the single entry point for all LLM access:
- `complexity="high"` → `settings.LLM_MODEL_COMPLEX` (default: `gpt-4o`)
- `complexity="standard"` → `settings.LLM_MODEL_STANDARD` (default: `gpt-4o-mini`)

Every node calls `get_llm()` at runtime — no model names are hardcoded in node files. Switching models requires only `.env` changes.

**Provider switching**

When `LLM_PROVIDER=anthropic`, `get_llm()` returns `ChatAnthropic` instead of `ChatOpenAI`. The `LLM_MODEL_COMPLEX` and `LLM_MODEL_STANDARD` values in `.env` should be changed to Anthropic model IDs (e.g., `claude-sonnet-4-20250514` / `claude-haiku-4-5-20251001`). No node code changes required — nodes call `llm.invoke(messages)` on whatever `BaseChatModel` they receive.

When `LLM_PROVIDER=mock`, `get_llm()` returns a `MockChatModel` — a full `BaseChatModel` subclass that detects which pipeline node is calling it (by pattern-matching keywords in the system prompt) and returns deterministic, realistic-looking JSON responses. This allows the entire pipeline to run end-to-end without any API keys — essential for CI testing and demos.

**Search provider abstraction**

`SearchClient` (`backend/search/base.py`) is an abstract base class with one method: `search(query, num_results) -> list[{url, title, snippet}]`.

- `SerperSearchClient`: Fully implemented. Uses `httpx.AsyncClient` to POST to Serper's Google Search API.
- `TavilySearchClient`: Stub — raises `NotImplementedError`. Interface defined, implementation not built.
- `YouSearchClient`: Stub — same.
- `MockSearchClient`: Fully implemented. Pattern-matches queries for known topics (Earth/Sun orbit, water boiling point, Great Wall from space) and returns pre-written realistic results. Falls back to generic Reuters/AP/Britannica results for unknown queries.

`get_search_client()` reads `settings.SEARCH_PROVIDER` and returns the appropriate implementation. The `search_node` calls this at runtime — no search implementation details leak into the node.

**Temperature = 0 on all calls**

Both `ChatOpenAI` and `ChatAnthropic` are instantiated with `temperature=0`. Deterministic, reproducible outputs — a verification pipeline should return consistent verdicts for the same input.

### 1.9 Docker Setup

**backend/Dockerfile**

```dockerfile
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt  # Layer cached separately from code
COPY backend/ ./backend/
COPY mcp/ ./mcp/       # MCP server shares the same backend.* imports
EXPOSE 8000
CMD ["uvicorn", "backend.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

Build context is the project root (set in docker-compose.yml `context: .`). This allows copying both `backend/` and `mcp/` while keeping the import structure intact.

**frontend/Dockerfile — multi-stage**

Stage 1 (build):
```dockerfile
FROM node:20-slim
WORKDIR /app
COPY frontend/package*.json ./   # Layer cached: only reinstalls when lockfile changes
RUN npm ci                        # Reproducible install from lockfile
COPY frontend/ .
RUN npm run build                 # Vite produces dist/
```

Stage 2 (serve):
```dockerfile
FROM nginx:alpine                 # Tiny final image
COPY --from=build /app/dist /usr/share/nginx/html
COPY frontend/nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 3000
```

The multi-stage build means the final image contains only nginx and the built static files — not Node.js or source code. Significantly smaller and leaner than a single-stage build.

**nginx.conf**

```nginx
server {
    listen 3000;

    location / {
        root /usr/share/nginx/html;
        index index.html;
        try_files $uri $uri/ /index.html;   # SPA fallback — React Router handles routing
    }

    location /api/ {
        proxy_pass http://backend:8000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;    # Required for WebSocket upgrade
        proxy_set_header Connection "upgrade";      # Required for WebSocket upgrade
        proxy_set_header Host $host;
        proxy_read_timeout 300s;                    # Matches HITL 5-minute timeout
    }
}
```

The `Upgrade` and `Connection` headers are required for the WebSocket handshake. Without them, nginx strips the upgrade headers and the WebSocket connection fails with 426. The `proxy_read_timeout 300s` matches the HITL timeout — nginx won't close the connection while the pipeline is legitimately paused waiting for user input.

`try_files $uri $uri/ /index.html` is the standard SPA pattern: serve the exact file if it exists, otherwise serve `index.html` and let React Router handle it client-side.

**docker-compose.yml**

```yaml
services:
  backend:
    build:
      context: .
      dockerfile: backend/Dockerfile
    ports: ["8000:8000"]
    env_file: [.env]
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/api/health"]
      interval: 10s
      timeout: 5s
      retries: 3

  frontend:
    build:
      context: .
      dockerfile: frontend/Dockerfile
    ports: ["3000:3000"]
    depends_on:
      backend:
        condition: service_healthy   # Frontend waits for backend to pass healthcheck
```

`depends_on: condition: service_healthy` is the critical line. Without it, the frontend might start serving before the backend API is ready — users would get connection refused errors. With it, Docker won't start the frontend until the backend has successfully responded to the health check three times. Eliminates the startup race condition.

### 1.10 Key Design Decisions Summary

These are the decisions worth explaining in a technical interview, with one-line rationale for each:

- **LangGraph for the pipeline** — stateful, conditionally-routed, async-native; graph structure matches the architecture diagram; self-documenting in a way a plain async chain is not

- **asyncio.Event for HITL instead of LangGraph checkpointing** — avoids refactoring the working Phase 2 invocation pattern; works cleanly because pipeline and WebSocket handler share the same event loop; textbook asyncio coordination with no external dependencies

- **Adversarial search — affirm + refute queries per claim** — prevents confirmation bias; finding contradicting evidence is what separates genuine fact-checking from confirmation-bias search dressed up as verification

- **Dual-model pattern — GPT-4o for reasoning, GPT-4o-mini for structure** — six calls requiring genuine comprehension use the stronger model; twelve structured calls use the cheaper model; ~$0.05 per run total cost

- **Domain heuristic source classification with no LLM call** — instant and free; directionally correct for most sources; limitation honestly documented and framed; ML classifier on v2 roadmap

- **`emit()` vs `aemit()` for thread safety** — sync nodes run in LangGraph's ThreadPoolExecutor, requiring `run_coroutine_threadsafe`; async nodes run in the event loop using direct `await`; mixing these up causes silent failures or crashes

- **In-memory run store** — eliminates Redis/database dependencies; appropriate for single-process deployment; keeps the architecture explainable without operational complexity

- **Per-run asyncio.Queue** — clean decoupling between pipeline execution (background asyncio task) and WebSocket delivery (separate async task draining the queue); no shared mutable state between the two

- **Sentinel object for queue EOF** — `_SENTINEL = object()` signals pipeline completion through the same queue as events; the `send_events()` coroutine exits cleanly on sentinel detection

- **ClaimModal cancel → auto-approves** — closing the modal doesn't abandon the pipeline; `App.jsx` passes `onClose={() => submitHitl(hitlClaims)}` so the pipeline always gets a response regardless of how the modal is dismissed

- **`currentRunRef` in useVerify** — prevents stale WebSocket events from a previous run from corrupting the UI state when the user re-submits before the old connection fully closes

- **`depends_on: condition: service_healthy` in docker-compose** — eliminates the startup race condition where the frontend launches before the backend API is ready

- **MCP two-step pattern for interactive verification** — replicates HITL's spirit within stateless protocol constraints; step 1 returns claims for review, step 2 takes approved IDs and runs verification

- **MockChatModel + MockSearchClient** — full pipeline runs end-to-end with `LLM_PROVIDER=mock SEARCH_PROVIDER=mock`; `MockChatModel` detects its calling context by pattern-matching system prompt keywords; enables testing and demos without real API keys

- **`temperature=0` on all LLM calls** — deterministic, reproducible outputs; a verification pipeline should return consistent verdicts for the same input and evidence

- **`proxy_read_timeout 300s` in nginx** — matches the HITL timeout; prevents nginx from closing the WebSocket while the pipeline legitimately waits for user input for up to 5 minutes

- **Multi-stage Docker build for the frontend** — final image contains only nginx and static files, not Node.js or source code; smaller image, smaller attack surface

- **Conditional `no_claims` edge after HITL** — handles the zero-approved-claims edge case gracefully; produces a clean "low" verdict with `total_claims=0` rather than crashing or hanging
